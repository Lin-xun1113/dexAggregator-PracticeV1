# 简易多 DEX 并发询价器 (Aggregator Router)

## 项目背景

这是一个用 Go 语言编写的练手项目，旨在通过模拟 Web3 中去中心化交易所（DEX）的聚合询价功能，帮助开发者熟悉 Go 的高并发处理机制（如 Goroutine, WaitGroup, Channel, Select）以及工程化目录拆分规范。

本项目最终对应 CEX 后端开发岗位中对于“高并发处理”、“微服务/解耦化开发”以及“链上交互”的核心要求。

## 目前已完成的 “脚手架”

为了不让初学者陷入建立空文件夹或者思考如何解耦的泥沼中，本项目已经提前铺设了以下核心框架：

- `model/quote.go`: 定义了统一的询价返回结构体 `QuoteResult`。
- `dex/interface.go`: 定义了所有 DEX 都必须遵守的接口标准 `GetQuote()`，实现真正的面向接口编程。
- `dex/mock_dex.go`: 一个模拟的 DEX（会随机睡眠模拟网络延迟，并返回随机价格），用于前期打通并发逻辑。
- `main.go`: 最核心的入口文件。目前使用了基础的 `sync.WaitGroup` 启动了 3 个 Goroutine 来并发查询 3 个 MockDEX，并比较出了最优报价。

## 🔴 [YOUR TURN] 你的练手任务 (Next Steps)

脚手架搭好后，真正的硬核逻辑需要你亲自动手完成！请从以下两个方向中选择一个开始你的练习：

### 挑战任务 A: Go 进阶并发与超时控制 (面试最高频)

目前的 `main.go` 使用 `WaitGroup.Wait()` 会“死等”所有的 DEX 返回结果。如果有一个 DEX 的 API 挂了，用户就会永远卡住。

- **你的任务：** 修改 `main.go`，废弃 `WaitGroup` 的死等机制。引入 `Channel` 和 `select` 语法，为其加上**超时控制** (Timeout Control)。
- **目标效果：** 设定一个最大等候时间（比如 500ms），如果时间到了还没返回的 DEX 结果就直接作废，只比较已经返回的结果。

### 挑战任务 B: 真实 Web3 数据接入 (实战真金白银)

目前的 DEX 是假的，返回的数字也是随机生成的。既然你是智能合约老手，我们需要去抓取真实价格。

- **你的任务：** 在 `dex` 目录下新建一个 `uniswap.go` 或 `binance.go`，让其实现 `DEX` 接口。
- **目标效果：** 在里面手写 Go 的 `net/http` 原生网络请求库（或者引入第三方 SDK），调用 Binance 的公开 K 线 API 或 Uniswap 的公开 Subgraph API，解析 JSON，并将真实的实时价格封装成 `QuoteResult` 返回给 main 函数。
